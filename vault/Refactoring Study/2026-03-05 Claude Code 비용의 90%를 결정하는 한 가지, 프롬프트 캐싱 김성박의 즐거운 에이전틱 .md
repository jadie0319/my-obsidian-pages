## 1. Highlights/Summary
이 글의 핵심은 Claude Code 비용 최적화의 중심이 프롬프트 캐싱 (prompt caching)이라는 점이다. 입력 토큰 과금이 비용의 대부분을 차지하는 구조에서, 캐시 읽기 비용은 일반 입력의 10분의 1 수준이어서 같은 작업도 캐시 적중 여부에 따라 비용이 크게 달라진다고 설명한다.

글은 Claude Code 요청이 `tools -> system -> messages` 순서로 누적되고, 접두사 매칭 (prefix matching) 기반으로 캐시가 동작한다는 구조를 풀어낸다. 즉, 상위 접두사가 안정적으로 유지되면 새 메시지 일부만 비싼 입력 토큰으로 계산되고, 나머지는 저렴한 캐시 읽기로 처리된다.

또한 실무 관점에서 모델 전환, 도구 토글, `/clear` 남용, TTL (time to live) 만료 같은 행동이 캐시를 깨뜨려 비용을 키운다고 경고한다. 결론적으로 세션 일관성 유지, 캐시 적중률 (cache hit rate) 모니터링, 상황별 세션 전략이 비용 절감의 핵심 실행 포인트로 제시된다.

## 2. Detailed Summary

### 2.1 프롬프트 캐싱의 비용 구조와 작동 원리
글은 Claude Sonnet 4.6 기준 가격을 예로 들어 일반 입력 `$3/MTok`, 캐시 쓰기 `$3.75/MTok`, 캐시 읽기 `$0.30/MTok`를 제시하며, 초기 1회 캐시 생성 이후 반복 요청에서 큰 절감이 가능하다고 설명한다. 100K 컨텍스트의 20턴 세션 시뮬레이션에서는 캐싱 미적용 약 `$6`, 캐싱 적용 약 `$1` 수준으로 비교한다.

중요한 메커니즘은 요청 “앞부분”의 동일성이다. 접두사 연속 구간이 같아야 캐시 히트가 발생하고, 중간/후반 일부가 같아도 효과가 없다는 점을 강조한다. 또한 최소 접두사 길이 조건(모델별 상이)이 있으며, 상위 레이어 변경이 하위 캐시 무효화로 이어지는 계층 구조를 함께 설명한다.

### 2.2 Thariq의 5가지 교훈: 사용자 행동이 곧 비용
글은 “정적 콘텐츠 먼저, 동적 콘텐츠 나중” 원칙을 반복해서 강조한다. 도구 정의, 시스템 프롬프트, CLAUDE.md, 세션 컨텍스트 순으로 안정성을 확보하고, 매 턴 바뀌는 새 메시지는 하단에 두어야 캐시 효율이 높다.

특히 세션 중 모델 전환(`/model`)과 도구/속도 토글(`/fast`, 웹 검색, 확장 사고)은 캐시를 크게 훼손하는 행동으로 정리된다. 모델 캐시는 모델별로 격리되어 전환 시 사실상 재시작되며, 시스템 프롬프트 변경은 하위 캐시를 연쇄 무효화한다.

또한 CLAUDE.md를 세션 중 변경하지 말고 시작 전에 고정하라는 실무 팁, 포크 작업(`--fork-session`) 시 부모 접두사를 유지해야 캐시를 공유할 수 있다는 운영 팁이 함께 제시된다.

### 2.3 실전 전략: 세션 유지, `/clear` vs `/compact`, TTL 리듬
글은 같은 파일군 작업은 한 세션에서 이어가라고 권한다. 세션을 나누면 캐시를 매번 다시 쌓아야 하므로 비용이 증가하고, 연속 작업(버그 수정 후 테스트 작성 등)은 동일 세션에서 처리할수록 유리하다고 본다.

`/clear`와 `/compact`의 차이도 명확히 구분한다. `/clear`는 히스토리 초기화로 캐시 재시작에 가깝고, `/compact`는 핵심을 남긴 압축으로 캐시 관점에서 상대적으로 유리할 수 있다. 다만 압축 결과에 따라 접두사가 바뀌면 미스가 날 수 있으므로 측정 기반 판단이 필요하다고 말한다.

TTL 기본 5분(플랜에 따라 1시간 옵션)을 기준으로, 장시간 중단 시 캐시 만료로 재작성 비용이 생긴다는 점도 다룬다. 그래서 작업 리듬 유지, 긴 휴식 전 컨텍스트 정리, 플랜 특성에 맞는 TTL 전략을 권장한다.

### 2.4 모니터링과 구현 디테일: 상태 라인, 캐싱 모드, 룩백 윈도우
글은 상태 라인 (status line)에서 `current_usage`의 토큰 필드를 활용해 캐시 적중률을 직접 계산하라고 제안한다. 제시된 계산식은 `캐시 히트율 = cache_read_input_tokens / (cache_read_input_tokens + cache_creation_input_tokens + input_tokens) * 100`이다.

원문에는 설정/의사코드 예시도 포함된다: `{"statusLine":{"type":"command","command":"~/.claude/statusline.sh"}}`, `ANTHROPIC_LOG=debug claude`, `client.messages.create(..., cache_control={"type":"ephemeral"}, ...)`. 자동 캐싱과 명시적 캐시 브레이크포인트를 함께 설명하며, 최대 4개 브레이크포인트와 20블록 룩백 윈도우 제한이 긴 대화의 캐시 효율 저하 요인이 될 수 있다고 정리한다.

불확실: 원문 추출본에서 코드 블록 줄바꿈이 일부 평탄화되어 스크립트 예시는 실행 코드라기보다 의사코드 형태로 해석하는 것이 안전하다.

## 3. Conclusion and Personal View
- 프롬프트 캐싱은 비용 최적화의 “선택 사항”이 아니라 “설계 기준”이다: 글 전체가 비용 차이를 수치로 보여주며 근거를 제공한다.
- 세션 일관성이 곧 비용 통제력이다: 모델/도구 설정을 자주 바꾸는 습관만 줄여도 불필요한 재캐싱을 크게 줄일 수 있다.
- `/clear`와 `/compact`를 구분하는 운영 습관이 필요하다: 둘 다 컨텍스트 정리 명령이지만 비용 영향은 다르다.
- 캐시 적중률을 관측하지 않으면 개선도 불가능하다: 상태 라인 지표를 보고 행동을 교정하라는 메시지가 실무적이다.
- “긴 대화일수록 싸진다”가 아니라 “접두사가 안정적일수록 싸진다”가 더 정확한 원칙이다: 구조 이해가 운영 판단의 품질을 높인다.
- 포크/병렬 작업도 캐시 관점에서 설계해야 한다: 동일 접두사 유지 여부가 병렬 처리 비용을 좌우한다.
- TTL은 기술 제약이 아니라 팀 작업 리듬과 연결된 비용 변수다: 휴식/전환 타이밍까지 운영 전략에 포함해야 한다.
- 실무자에게 이 글이 중요한 이유는 간단하다: 같은 생산성을 유지하면서 토큰 비용과 응답 효율을 동시에 개선할 수 있는 구체적 행동 지침을 제공하기 때문이다.
